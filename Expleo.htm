Feature

I primarily worked on User Authentication for the Hands-Free Module (HFM) and the Body Control Module (BCM), focusing on automotive body features rather than mechanical components. These features were centered around functionality.

User Authentication System

This module was responsible for unlocking or locking the car based on various authentication methods. While it may seem straightforward, its complexity increases with different scenarios, such as multiple authentication options—physical keys, transponders, and mobile apps for rental vehicles. Each method requires careful handling of conditions and encryption. A key aspect of this work involved preventing relay attacks , ensuring the system's integrity against unauthorized access. This was primarily designed using Stateflow in a Simulink model.

Driver Input Management

This module managed the transition between manual and automatic driving modes in certain vehicles, such as sports mode, eco mode, and normal mode. It included state definitions and transitions based on various conditions. I was responsible for the maintenance and debugging of this module as part of the development process.

Both features were integral to the Renault-Nissan architecture and encompassed various car lines.

Relay attacks in automotive security are a type of unauthorized access where an attacker intercepts and relays signals between a key fob and a vehicle. The process involves the attacker capturing the signal from the key fob—often located inside the owner's home—using one device, and then transmitting that signal to the vehicle with another device, tricking it into believing the key fob is nearby. This allows the attacker to unlock and start the car without the owner’s knowledge. To mitigate the risk of relay attacks, vehicle owners can use signal-blocking pouches for their key fobs and manufacturers can implement enhanced security features in vehicles.

Toolchain Process Flow

1. Requirement Analysis via DOORS and Error Ticket Review
Objective: Analyze requirements in DOORS and investigate error tickets from SIL and hardware testing teams.
Activities:
  • Receive requirement details in DOORS.
  • Analyze error tickets from the SIL or hardware testing teams.
  • Assess whether software changes are feasible or if another team is responsible.
2. Development Phase and Team Communication
Objective: Implement changes in the model based on requirement analysis.
Activities:
  • Modify the model based on the requirements.
  • Understand and explain the changes to the team lead.
3. MAAB Check Based on MXAM Rules
Objective: Ensure the code architecture complies with MAAB guidelines using MXAM.
Activities:
  • Run MXAM checks on the modified model.
  • Ensure compliance with MAAB rules.
4. Embedded Code Development and GUI Integration
Objective: Develop embedded C code and create an s-function for GUI integration.
Activities:
  • Generate embedded C code.
  • Create an s-function for model integration.
  • Develop a standalone GUI for feature functionality and testing.
5. Model and GUI Verification by MIL Testing Team
Objective: Send the updated model and GUI to the architect and MIL testing team for verification.
Activities:
  • Send the model and GUI for functional verification.
  • Receive feedback from the architect and testing team.
  • Make necessary rectifications based on feedback.
